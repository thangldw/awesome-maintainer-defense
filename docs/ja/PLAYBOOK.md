# メンテナー防御プレイブック

## レベル0 — ベースライン

- コントリビューション、AI支援、未依頼PR、行動規範、セキュリティ報告のポリシーを公開する。
- バグ報告にバージョン、環境、最小再現手順、実際の結果、期待結果を必須化する。
- `.github/`、リリースワークフロー、パッケージマニフェスト、所有者ファイルをCODEOWNERSと必須レビューで保護する。
- rulesetでレビュー、ステータスチェック、削除防止、force-push防止を設定する。
- ジョブごとに`GITHUB_TOKEN`権限を明示し、サードパーティActionをレビュー済みコミットSHAに固定する。
- private vulnerability reportingを有効化し、インシデント制御を有効化できる担当者を決める。

## レベル1 — 観測

- 提出数、初回応答時間、クローズ理由、再オープン数、レビュー時間を測定する。
- 新しいツールを少なくとも代表的な2レビュー周期、dry-run/report-onlyで動かす。
- 誤検知だけでなく見逃しも確認するため、フラグあり・なしの両方を標本レビューする。
- コントリビューター種別、言語、コード領域、理由ごとに誤検知を記録する。
- スコアだけで悪意やAI利用を断定しない。

問題、週次コスト、有効なシグナル、許容誤検知率をチームが説明できるまで次へ進みません。

## レベル2 — Review-first

- `needs-human-review`のような中立的ラベルを使用する。
- 自動close、lock、block、公開の非難は無効にする。
- メンテナー、承認済みbot、既知のコントリビューター、重要なリリース自動化だけを理由付きで除外する。
- 高影響操作には必ず人間の判断を要求する。
- 新しい証拠に基づく短い異議申立て経路を公開する。

[Balanced starter kit](../../kits/balanced)はPRトリアージでこのレベルを実装します。

## レベル3 — インシデントモード

1. インシデント所有者を任命し、タイムスタンプ付き判断ログを開始する。
2. URL、スクリーンショット、Webhook delivery ID、workflow run ID、監査ログを保存し、シークレットは記録しない。
3. 完全性が不明な場合、危険な自動化とリリースを一時停止する。
4. existing users、contributors、collaboratorsに一時的なinteraction limitを設定する。
5. 影響を受ける提出だけをclose/lockし、広範な制御には有効期限を設定する。
6. 技術的な意見の相違ではなく、プラットフォーム規則違反に対してblockまたはabuse報告を行う。
7. 漏えい認証情報をローテーションし、成果物を無効化し、侵害の可能性があればワークフロー履歴を確認する。
8. 嫌がらせを拡散せず、機密証拠を公開しない短いステータスを通知する。

## レベル4 — 復旧

- 予定どおりinteraction limitとlockdownを解除する。
- 広範な制御に巻き込まれた正当な項目を再オープンする。
- 誤検知で公開操作が行われた場合、影響を受けたコントリビューターに通知する。
- レビュー負荷、見逃した正当な報告、誤検知、復旧時間をベースラインと比較する。
- 実証された制御だけを長期ポリシーにする。
- 攻撃者の行動、プラットフォーム制約、設定ミス、メンテナー能力を分けた責任追及のない振り返りを行う。

各自動化には、所有者、測定可能な目的、権限、データ境界、高影響操作、レビュー日、ロールバック手順が必要です。どれかが欠ける場合は執行モードを継続しないでください。
