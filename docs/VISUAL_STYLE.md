# Visual diagram guidelines

Diagrams in this repository should shorten a decision or reveal a relationship that prose does not make obvious. They are not decoration.

## Rules

1. Give each diagram one explicit purpose and a bounded scope.
2. Prefer standard flowchart notation: terminal for a start or end, rectangle for an action, diamond for a decision, and labeled arrows for outcomes.
3. Keep typography, node shapes, arrow direction, and terminology consistent. A style change must encode meaning.
4. Show only essential decisions. Split a crowded diagram into smaller diagrams instead of compressing more detail into one canvas.
5. Group related concepts spatially and keep a clear reading direction, normally top-to-bottom for repository documentation.
6. Use color only for stable semantic categories or status. Every meaning must also be expressed by text or shape.
7. Keep node labels short, use descriptive link text, and provide the equivalent diagram in English, Vietnamese, and Japanese when it changes adoption decisions.
8. End operational diagrams in a concrete action, owner decision, or measurable next state.

These rules adapt Miro's guidance to start with a clear purpose, keep flowcharts simple and scalable, use standard symbols, group related ideas, and maintain visual consistency. See Miro's [flowchart guide](https://miro.com/flowchart/how-to-make-a-flowchart/) and [diagram design principles](https://miro.com/blog/diagram-design/).
